Panic! In The Toolshed

(Work in progress)

Matt Dray, June 2023

tl;dr

  1. Do less
  2. Iterate, then iterate again
  3. Make things open, it makes things better
  • ‘Too long didn’t read’.
  • The three main takeaways are stolen from the Government Design Principles.
  • ‘If we’ve found a way of doing something that works, we should make it reusable and shareable instead of reinventing the wheel every time.’
  • ‘The best way to build good services tools is to start small and iterate wildly.’
  • ‘Share code, share designs, share ideas, share intentions, share failures.’

A cautionary tale

  • I think I’m here to share some wisdom.
  • The best example I have is an R package I wrote that has mass appeal.
  • I want to tell you what worked and what I regret.

Thought process (circa 2021)

  1. There’s new spreadsheet guidance
  2. Might as well write some functions
  3. Other people are probably doing this
  • Context: the Government Analysis Function created new (and very good) guidance for releasing statistics in spreadsheets.
  • None of the spreadsheets in my team were compliant. Colours, blank columns, multiple headers. Classic non-accessible stuff. Also the whole process was point-and-click before I joined.
  • Given multiple files staggered across releases, it made sense to create some reusable R functions to apply the new guidance consistently, which in turn meant it was worth while to create a package.
  • Att the time, there was the Python package ‘gptables’, but it hadn’t yet been updated for the new guidance (it has been now) and there was no native R solution.

  • And so the {a11ytables} package was created. Entirely for our own use at first, but I got the feeling it would have other interested parties.
  • What does it do? You give it your tables of data, along with some metadata like worksheet titles, and it automatically creates an xlsx spreadsheet that is (mostly!) compliant with the guidance.

Success!

  • It did what I wanted it to do
  • I did some (non-spammy?) promotion
  • It’s in use across government
  • I used it to successfully create the spreadsheets we needed to release. The lowest bar!
  • I spoke to some targeted groups that might find it useful: bot internally (i.e. at an NHS-R webinar, at a GSS Champions quarterly meeting, to a Scottish Government R conference) and externally (i.e. a talk at EARL conference 2022, blog posts, social media). It’s referenced now from the Analysis Function spreadsheet guidance.
  • It’s not perfect, but the package is now in use across a number of organisations.

Uhoh…

  • The (only) maintainer left the org!
  • Should {a11ytables2} exist?
  • Where should these things live?
  • Maybe the {a11ytables} package could change GitHub organisation? Need to think about licensing, broken URLs, burden on new owner.
  • Could ask Cabinet Office to archive the original {a11ytables} and clearly signpost this in the README, then communicate widely about the new package and target known users.
  • Another example is the {fingertipsR} package, which was removed from CRAN. URLs have broken, search engines show the ‘removed from CRAN’ page high in the rankings.

A resolution

1. Think sustainably

  • Build modular things
  • Write good documentation
  • Plan for your loss

2. Advertise

  • You need momentum
  • Show it off
  • Answer questions, encourage collaborators

3. Centralise

  • Work together, avoid duplication
  • Encourage institutional ownership
  • Build a toolshed, they will come?
